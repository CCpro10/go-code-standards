# Go 项目规则

本文件是本 Skill 的最高优先级规则，必须完全遵守。英文版本必须与本文件保持同步；当英文版本、通用 Go 风格规则或本地习惯与本文件冲突时，以本文件为准。

这些规则来自用户在实际 Go 业务代码中的要求，目标是保持边界清晰、失败显式、逻辑短直、代码命名和结构可读。

## 优先级

1. 本文件规则：最高优先级，必须完全遵守。
2. `references/go-style-rules.zh.md`：来自 Google Go Style 和 Uber Go Style Guide 的通用 Go 风格规则。
3. 仓库本地约定：在不违反前两类规则时遵守。

## 边界和失败

- 遵循契约式设计：调用方保证前置条件，函数保证后置条件。函数契约中已经要求合法的输入，函数内部不要再用兜底分支吞掉非法输入。
- 遵循 Fail Fast：错误越早暴露越容易定位。业务代码应让 bug 在离源头最近的地方失败，而不是在后续流程里被默认值、过滤或 normalization 吃掉。
- 不要在业务方法里保留“理论上不会发生”的分支。依赖缺失、配置缺失、构造参数错误应在启动或构造期直接失败，不要让每个 RPC 方法返回 `not implemented` 这类运行时兜底响应。
- 一段逻辑不要试图兼容过多来源不明的情况。缺字段、状态不合法、依赖未初始化、调用顺序错误应尽早暴露为清晰错误，而不是继续向后执行。
- 可降级能力必须是明确设计出来的业务能力；如果只是为了避免报错而兜底，优先改成显式失败。
- 信任调用的函数边界。如果被调用函数可能返回不合法结果，应在被调用函数内部返回错误，而不是在外层无脑兜底检查。
- 对关键失败路径记录有意义的日志，并包装错误，让调用链能看到业务语境。

## 输入、默认值和规范化

- 不要用含糊的 `normalizeXxx` 函数隐藏 trim、过滤、去重、默认值补齐和数据重塑。函数名必须暴露真实操作，例如 `validateSkillRefs`、`dedupeSkillRefs` 或 `parseSkillRefs`。
- 不要让系统同时接受合法和非法业务输入。必填业务字段必须在请求解析、构造期或 service 入口校验，非法时清晰失败，不要在深层业务逻辑中静默修复。
- 遵循 Parse, Don’t Validate：不要在系统各处反复判断同一份数据是否合法；应在入口处把不可信输入解析成可信类型，进入核心业务后默认它已经满足契约。
- 尽量让非法状态不可表示。对关键业务字段，优先通过构造函数、私有字段、包边界或专门类型表达已验证状态，例如用 `SkillKey` 类型承载非空且合法的 skill key，而不是到处传裸 `string`。
- 不要用含糊默认值掩盖调用链。请求字段如果是业务语义的一部分，例如 `occurred_at`，必须由调用方显式传入并在入口校验；不能在 service 内静默回退到 `time.Now()`。
- 业务配置优先显式声明。除非接口契约、产品规则或技术方案明确要求默认值，否则不要在代码里用默认值补齐缺失配置。
- 默认值只允许用于非业务语义的工程参数，或文档明确规定的兼容行为。补齐位置要靠近入口校验，并让字段来源、默认口径和适用范围一眼可见。
- 除非跳过是明确业务规则，否则不要用 `continue` 丢弃非法业务数据。对 `SkillKey` 这类必填字段，优先返回错误，而不是静默过滤。
- 当校验和转换都重要时，拆清楚两者。执行校验的函数不应同时改变业务语义，除非函数名和返回类型让这个契约非常明确。

## 构造和局部变量

- 空结构体和 nil 返回值优先显式直接返回。例如失败时返回 `dailyLoginRewardConfig{}`，不要提前声明一个空变量只为了失败路径复用。
- 在一个函数里，如果一个变量可能因为后续的失败而没有被使用，那么它不应该在此处声明，应该在失败点之后再声明。
- 不要先构造一个“默认业务对象”，再在条件分支里覆盖成另一种语义。业务对象应从明确输入解析出来，缺输入就失败，避免无效命名打断代码逻辑。
- 大业务结构体优先使用 keyed composite literal 一次性构造，让完整对象形状集中出现在一个地方。除非 API 或控制流确实需要可变赋值，不要先声明空结构体再逐字段赋值。

## 函数、文件和拆分

- 好代码优先短而直。能用少量顺序代码表达清楚的逻辑，不要拆成大量单次使用的小函数，也不要为了“消除分支”引入冗长的配置归一化层。
- 不要为了只使用一次的逻辑额外抽小函数。只有当函数能降低复杂度、表达明确业务概念或被多处复用时才抽出来。
- 不要把代码切成过多小函数；优先按职责拆文件，再保留少量有业务含义的函数。
- 函数代码必须包含注释。简单函数至少说明业务意图、输入边界或返回语义；复杂函数必须有更详细的注释，说明关键分支、失败边界、状态变化或业务约束。
- 注释必须提供信息量，不写“给变量赋值”“调用方法”这类只复述代码的注释。
- 关键业务逻辑需要配合有意义的日志说明。
- 使用合理的空行和换行让逻辑分段清晰。

## 结构体

- 结构体必须必要、清晰，并且能降低理解成本。禁止为了中间转换、临时拼装或掩盖调用链而定义大量中间结构体。
- 定义结构体前先确认它是否表达稳定业务概念、外部契约、持久状态或多处复用的数据形状。没有这些必要性时，优先使用明确输入、局部变量或直接构造目标对象。
- 结构体会增加阅读和维护成本。字段含义、来源和生命周期不清楚的结构体不要定义。

## Package 和命名

- 在每个 package 中，最重要的可导出 package-level 函数和方法必须放在最前面，其次是其他可导出函数和方法，最后才是不可导出的函数和方法。
- 将第一个不可导出的函数或方法视为 private section 的开始；其后不要再放大写开头的函数或方法。
- 禁止把非 API 函数标注为可导出。不作为 package API 的函数必须以小写字母开头，不能为了跨文件调用、测试方便或看起来重要而使用大写开头。

## 指针 helper

- 不要手写 `stringPtr`、`int64Ptr`、`boolPtr` 这类指针 helper；统一使用 `code.byted.org/lang/gg/gptr` 的 `gptr.Of(...)`。
- 常量数字传给 `gptr.Of` 时注意类型推断；需要 `*int64`、`*int32` 等明确类型时使用 `gptr.Of[int64](...)`、`gptr.Of[int32](...)`。
