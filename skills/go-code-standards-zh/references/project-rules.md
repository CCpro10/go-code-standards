# Go 风格与可维护性规则

本文件是 Go style Skill 的最高优先级规则，必须完全遵守。它只处理风格、可读性和可维护性问题，默认代码逻辑是正确的。

不要用本 Skill 查并发问题、可疑 bug、性能问题或业务逻辑错误；这些风险项交给 `code-risk-review`。

## 优先级

1. 本文件规则：最高优先级。
2. `references/go-style-rules.md`：来自 Google Go Style 和 Uber Go Style Guide 的通用 Go 风格规则。
3. 仓库本地约定：不违反前两类规则时遵守。

## 目录和 Package

- 目录层级应表达业务模块和职责边界，不要按临时实现细节、工具函数或过深技术分层切目录。
- package 名称要短、小写、有意义；避免 `util`、`common`、`misc` 这类无法表达职责的名字。
- 一个 package 内最重要的可导出函数和方法放在最前面，其次是其他可导出函数和方法，最后是不可导出函数和方法。
- 将第一个不可导出函数或方法视为 private section 的开始；其后不要再放大写开头的函数或方法。
- 禁止把非 API 函数标注为可导出。不能为了跨文件调用、测试方便或看起来重要而大写开头。

## 结构体

- 结构体必须必要、清晰，并且降低理解成本。禁止为了中间转换、临时拼装或掩盖调用链而定义大量中间结构体。
- 定义结构体前先确认它是否表达稳定业务概念、外部契约、持久状态或多处复用的数据形状。
- 如果只是单次传递几个字段、临时组装参数或给后续覆盖使用，优先使用明确输入、局部变量或直接构造目标对象。
- 字段含义、来源和生命周期必须清楚；看不出边界的结构体不要定义。

## 函数和方法

- 函数和方法的划分必须合理。方法应表达接收者的行为或状态变化；如果和接收者没有真实关系，优先使用普通函数。
- 好代码优先短而直。能用少量顺序代码表达清楚的逻辑，不要拆成大量单次使用的小函数。
- 不要为了只使用一次的逻辑额外抽小函数。只有当函数能降低复杂度、表达明确业务概念或被多处复用时才抽出来。
- 不要把代码切成过多小函数；优先按职责拆文件，再保留少量有业务含义的函数。
- 函数代码必须包含有信息量的注释。简单函数说明业务意图、输入边界或返回语义；复杂函数说明关键分支、状态变化或业务约束。
- 注释不能只复述代码，不写“给变量赋值”“调用方法”这类无信息量注释。

## 局部可读性

- 变量尽量延后声明。如果一个变量可能因为后续步骤失败而没有机会使用，就把声明放到失败点之后。
- 空结构体和 nil 返回值优先显式直接返回。例如失败时返回 `dailyLoginRewardConfig{}`，不要提前声明空变量只为失败路径复用。
- 大业务结构体优先使用 keyed composite literal 一次性构造，让完整对象形状集中出现在一个地方。
- 不要先声明空结构体再逐字段赋值，除非 API 或控制流确实要求可变赋值。
- 使用合理空行和换行表达逻辑阶段，让代码一眼能看出主流程。

## 命名和边界表达

- 命名必须让读者知道函数真实做了什么。避免用 `normalizeXxx` 隐藏 trim、过滤、去重、默认值补齐和数据重塑等多种行为。
- 如果确实需要转换、去重、解析或校验，函数名应明确表达真实动作，例如 `parseXxx`、`dedupeXxx`、`validateXxx`。
- 业务对象应从明确输入构造出来，不要先构造“默认业务对象”，再在分支里覆盖成另一种语义。

## 指针 Helper

- 不要手写 `stringPtr`、`int64Ptr`、`boolPtr` 这类指针 helper；统一使用 `code.byted.org/lang/gg/gptr` 的 `gptr.Of(...)`。
- 常量数字传给 `gptr.Of` 时注意类型推断；需要 `*int64`、`*int32` 等明确类型时使用 `gptr.Of[int64](...)`、`gptr.Of[int32](...)`。
